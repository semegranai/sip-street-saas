🔧 What Should Be Inside DESIGN-PHILOSOPHY.md

Here’s the simplified but powerful version you should include:

Core Principles
1. Speed Over Everything (But Never at the Expense of Trust)
Fast inputs
Instant feedback
No lag in decision-making
2. Trust is Non-Negotiable
Numbers must always match
No hidden calculations
Consistency across UI and PDF
3. Opinionated > Flexible
Default assumptions preferred
Limit unnecessary customization
Guide the user toward “correct” modeling
4. Simplicity Wins
Fewer inputs > more inputs
Clear layout > feature density
Remove anything confusing
5. One Source of Truth
No duplicated logic
No “slightly different” calculations
Everything flows from the same engine
6. Investor-First Output
PDF quality is critical
Clean presentation = perceived credibility
Output matters as much as math
7. Build for Scale, Not Just Today
Every feature should work across asset classes
Avoid hardcoding assumptions
8. Reduce Cognitive Load
User should not have to “figure things out”
System should guide decisions
⚡ The Real Role of This Doc

Every time a decision comes up, the team should ask:

“Is this making it faster or slower?”
“Is this increasing or decreasing trust?”
“Are we adding flexibility or clarity?”

If it violates the philosophy → don’t build it

🧭 Final Take

You were right to call this out.

If you get one thing right early, it’s this:

Most products fail because they slowly lose their philosophy.

Lock this doc in early, and everything else (UI, features, architecture) stays aligned.
